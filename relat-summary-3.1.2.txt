Why UsernameRule
LengthRule (3.1.1) is mostly one numeric input. Pairwise fits rules with severalconfiguration flags that interact. UsernameRule checks if the password matches the
username (forward) and optionally the reversed username when matchBackwards=true
Username in all tests: "admin" (Passay builds reversed as "nimda" via user.reverse())


Configuration factors
  | F1 matchBackwards | true, false |
  | F2 ignoreCase     | true, false |
  | F3 matchBehavior  | StartsWith, EndsWith, Contains |

Exhaustive on config only: 2×2×3 = 12 combinations
Pairwise on config: 9 tests (covers all 16 pairs of levels between F1, F2, F3)

Test input (not a 4th factor)
Each row picks a password on purpose to exercise:
  forward  — substring "admin" in the sense required by F3
  reversed — substring "nimda" (admin reversed), relevant when F1=true
  neither  — no "admin" / "nimda" match for the active checks

The 36 number in the Java comment is 2×2×3×3 if you also cross config with those three password goals -> we use 9 config rows and choose passwords per row
