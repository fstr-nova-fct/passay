Why NumberRangeRule:
Password must not contain a number in [lower, upper) — here [10, 20)
Rule also depends on how the number appears (Contains / StartsWith / EndsWith) and whether to report all matches or only the first

Specification:
NumberRangeRule(10, 20, matchBehavior, reportAllFailures)
Range [10, 20): lower inclusive, upper exclusive 
Fail → ILLEGAL_NUMBER_RANGE when a forbidden number matches with the given behavior

Category partition
  | Category |          Choices |
  | Forbidden number |  none, below(9), above(20), in-range 10/15/19, multiple(12+17) |
  | Digit position |    start, middle, end |
  | Match behavior |    Contains, StartsWith, EndsWith |
  | Failure reporting | report all, report first only |

Rationale:
They map to real rule inputs: which integer is present, where it sits in the password, MatchBehavior on the rule, and reportAllFailures 
TSL expands combinations; constraints remove useless ones

TSL constraints (NumberRangeRule.tsl):
->ends with only for in-range 10, 15, or 19 — else Contains / StartsWith only
->report first only when multiple (12+17); report all for all other number types

Full cross-product before constraints: 7×3×3×2 = 126
After constraints (verified by enumeration): 51 test frames

->in-range 10, 15, 19: 3 types × 9 each (3×3, report all only) = 27
